  🔧 Gaps — your to-do list, worst first

  [!]  #1  Colored diff output on assert_eq/ne failure       table-stakes · effort ~1d · confidence: high
       Why:    Every modern Rust asserter (pretty_assertions, k9, assert2, googletest) ships
               readable colored diffs; assertables emits a plain text dump. This is the single
               biggest UX gap vs. the field's baseline.
       Study:  rust-pretty-assertions/rust-pretty-assertions → src/lib.rs (Comparison::new)
       Step 1: Add `pretty_assertions = { version = "1", optional = true }` to Cargo.toml and a
               `pretty` feature. In src/assert_eq.rs, gate a new failure path that calls
               `pretty_assertions::Comparison::new(&a, &b)` and embeds its output in the panic
               message; keep the existing path as fallback when the feature is off.

  [!]  #2  Float ULPs comparison                              table-stakes (partial) · effort ~2h · confidence: high
       Why:    `assert_approx` covers absolute and relative error but not ULPs, the standard
               IEEE-754 comparison the `approx` crate (108M downloads) makes canonical.
               Without it, low-level numeric tests must reach for a second crate.
       Study:  rust-num/approx → src/ulps.rs (UlpsEq trait)
       Step 1: Create src/assert_approx/assert_approx_eq_with_ulps.rs containing
               `assert_approx_eq_with_ulps!(a, b, ulps)` and its `_as_result` twin, modeled on
               assert_approx_eq_with_absolute_error.rs. Add the module to src/assert_approx/mod.rs.

  [~]  #3  Non-fatal `check!` / multi-failure mode             edge · effort ~3d · confidence: medium
       Why:    assert2's `check!` and googletest's `expect_that!` let a test collect every
               failure instead of stopping at the first — a meaningful workflow improvement
               for snapshot-style tests with many independent comparisons.
       Study:  de-vri-es/assert2-rs → src/check.rs (the failure-aggregator)
       Step 1: Add src/check.rs exporting `check_*!` variants of the top-tier macros that push
               to a thread-local Vec<String> instead of panicking, with a `#[test]` harness
               helper that drains and reports at end-of-test.

  [~]  #4  no_std support                                      edge · effort ~2d · confidence: medium
       Why:    claim and static_assertions both support no_std; assertables's `format!()` in
               every macro forces `alloc`. Locks out embedded users — a niche but real audience.
       Study:  svartalf/rust-claim → src/lib.rs (no_std-compatible message builder)
       Step 1: Add `#![cfg_attr(not(feature = "std"), no_std)]` to src/lib.rs with a `std`
               default feature; swap `format!` for `core::fmt::Write` into a stack/heapless
               buffer in src/assert.rs first as a proof-of-concept, then propagate.

  [~]  #5  Drop-in std `assert!` replacement                   edge · effort ~1d · confidence: low
       Why:    assert2 and pretty_assertions both pitch themselves as "just import and your
               existing `assert!` calls get richer output." assertables requires renaming.
               Closing this lowers adoption friction.
       Study:  de-vri-es/assert2-rs → README ("drop in replacement for std::assert!")
       Step 1: Add a `prelude` module that re-exports a custom `assert!` macro_rules! using the
               same name as std — document the `use assertables::prelude::*;` pattern.
